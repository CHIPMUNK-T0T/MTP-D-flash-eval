## 社内文書管理システム v2.3 仕様概要

### システム概要
本システムは全国15拠点・従業員3,200名が利用する社内文書管理基盤です。2018年に初期導入し、5回の機能追加を経て現在のv2.3に至ります。主要機能は文書登録・検索・承認ワークフロー・版管理・アクセス制御の5つです。

### 技術スタック
フロントエンド: AngularJS 1.8（2022年EOL）
バックエンド: Java 8 / Spring Boot 2.3（2023年8月EOL）
データベース: Oracle 12c（2022年7月EOL済み）
ファイルストレージ: オンプレミスNAS（NetApp FAS2750）、総容量60TB、現在使用率83%
認証基盤: 独自LDAP連携（Active Directory非対応）
監視: Zabbix 4.0（サポート終了済み）
CI/CD: なし（手動デプロイ）

### パフォーマンス問題
全文検索の平均応答時間は15秒（目標値は5秒以内）です。ピーク時間帯（9:00〜10:00）には30秒を超えることがあります。月次バッチ処理（毎月末深夜）が翌朝の通常業務と重複した場合、過去6ヶ月で3回、最大2時間のシステム停止が発生しています。データベースのインデックス断片化率は平均67%で、週次のインデックス再構築に3〜4時間かかっています。

### セキュリティリスク
脆弱性スキャンにより、Java 8とOracle 12cに対してCVSS v3スコア8.0以上の未適用パッチが合計14件確認されています。外部アクセスはVPN経由ですが、多要素認証（MFA）は管理者アカウントのみに適用されており、一般ユーザーはパスワード認証のみです。アクセスログの保存期間は現在90日ですが、社内セキュリティポリシーおよびコンプライアンス要件は3年保存を義務付けています。直近のペネトレーションテストでは中リスク以上の脆弱性が8件検出されました。

### 運用・保守の問題
初期開発エンジニア4名のうち3名が退職済みで、仕様書・設計書の文書化が不十分な状態です。ソースコードのバージョン管理はSubversionで行っており、Gitへの移行は計画のみで未着手です。テスト環境が1台しかなく、本番環境との構成差異が10箇所以上あります。障害発生時の平均復旧時間（MTTR）は4.2時間で、直近12ヶ月の累計ダウンタイムは38時間に達しています。

### 拡張性の問題
NASの残余容量は約10TBです。データ増加ペースは年間8TBで、このままでは1年以内にストレージが枯渇します。データベース表領域の使用率は70%を超え、クエリプランが劣化しています。全文検索インデックスの再構築に平均4時間かかるため、スキーマ変更が事実上困難な状態です。

### 予算・体制
2025年度システム保守予算は1,200万円で、うち750万円がOracle・NetApp等のライセンス費に充当済みです。可用な開発予算は450万円です。社内IT部門は8名（インフラ3名、開発2名、ヘルプデスク3名）で、外部ベンダー1社と月次保守契約（月50万円）を締結しています。

### 直近インシデント

2025年3月: OracleDBのアーカイブログ領域が満杯となり全体停止（45分）。
